Architecture Proposal

Two approaches for structuring the video pipeline

Decisions So Far
Style: Motion graphics (no cursor)
Visuals: AI-generated images
Image providers: Multi-provider (adapter pattern)
Renderer: Remotion (React/TS)
Handoff: JSON manifest
Output: 1080p, up to 5 min

Approach A: Monorepo with packages/ directory

Remotion project lives inside fastpub-py

fastpub-py/
  fastpub/                  # Python pipeline
    pipeline/
      analyze.py
      scene_script.py       # NEW: LLM → scene script JSON
      image_gen.py          # NEW: multi-provider image generation
    render/
      video.py              # Orchestrator: calls stages → shells out to Remotion
    ai/
      tts.py                # Existing TTS
      image_providers/      # NEW: adapter pattern
        base.py             #   AbstractImageProvider
        xai.py              #   xAI Grok adapter
        seedance.py         #   Seedance adapter (future)

  packages/
    remotion-video/          # Remotion project (Node.js)
      src/
        Video.tsx            # Root composition
        scenes/              # Scene components (Hook, Problem, Results, etc.)
        animations/          # Shared animation primitives
      package.json
      remotion.config.ts

  tmp/                       # Build artifacts
    {job-id}/
      manifest.json          # Python → Remotion contract
      audio/                 # TTS audio files
      images/                # Generated images
      output.mp4             # Final video

Pros

  • Single repo, easy to version together
  • manifest.json is a local file, simple handoff
  • CI/CD in one place

Cons

  • Mixed Python + Node.js tooling
  • Users must install both ecosystems

Approach B: Separate Remotion package (npm)

Remotion lives in its own repo, installed as a tool

fastpub-py/
  fastpub/
    render/
      video.py              # Shells out: npx @fastpub/remotion-video render ...

fastpub-remotion/            # Separate repo / npm package
  src/
    Video.tsx
    scenes/
    animations/
  package.json

Pros

  • Clean separation of concerns
  • Remotion project independently versioned
  • Can be reused outside fastpub

Cons

  • Two repos to maintain
  • Manifest schema must stay in sync manually
  • More complex setup for contributors
Recommendation: Approach A (Monorepo)
The Remotion project is tightly coupled to fastpub's manifest schema — they evolve together. A monorepo keeps this coupling explicit and avoids sync issues. The mixed tooling cost is manageable with a simple make setup that installs both Python and Node deps.
