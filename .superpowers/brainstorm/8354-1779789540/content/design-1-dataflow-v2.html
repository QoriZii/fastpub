Design Section 1 (Revised): Data Flow

Scene script + image gen belong to slides; TTS + Remotion belong to video

REVISED PIPELINE
SLIDES STAGE (already exists, extended)
Scene Script
LLM → scenes[]
→
Image Gen
AI visuals per scene
→
HTML Slides
existing renderer
Output: slides HTML + scenes.json + images/ directory
↓
Video stage consumes slides output
VIDEO STAGE (new)
TTS Narration
audio per scene
→
Build Manifest
scenes + audio + images
→
Remotion Render
npx remotion render
Input: slides output • Output: output.mp4

What This Changes

Slides renderer gets richer
The existing render_slides() gains scene scripting and AI image generation. Slides become the source of truth for content — both the HTML deck and video consume the same scenes + images.
Video renderer is a thin layer
render_video() reads the slides output, adds TTS audio + timing, builds the manifest, and shells out to Remotion. No content generation — just audio + animation.
Shared content, two outputs
fastpub render -f slides → HTML deck
fastpub render -f video → runs slides stage first, then adds audio + renders MP4
fastpub render -f slides,video → slides stage runs once, both outputs produced
Manifest still the same
The JSON manifest schema doesn't change — it just gets assembled from slides output + TTS results instead of being built from scratch in the video stage.
Does this split feel right? Slides own the content (scenes + visuals), video just adds audio and renders.
